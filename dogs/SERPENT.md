# The Serpent Eating Its Own Tail

> *"The strange loop phenomenon occurs whenever, by moving upwards (or downwards) through the levels of some hierarchical system, we unexpectedly find ourselves right back where we started."*
> -- Douglas Hofstadter, *Godel, Escher, Bach* (1979)

> *"The serpent that bites its own tail is the oldest known mythical symbol... it represents the eternal return and the self-referential nature of life's code."*
> -- Jeremy Narby, *The Cosmic Serpent* (1998)

Strangen is self-hosting: the core library's `strange__space.h` is generated by the same pipeline it defines. The prototype declares types like `space`, `abstraction`, and `operation`; the parser and transformation engine use those types to read the prototype and produce the generated header; and that generated header is what makes the types available in the first place.

This circular dependency -- the serpent eating its own tail -- means that changes to the code generator must be introduced in a careful order to avoid breaking the bootstrap.

## The Bootstrap Cycle

```
strange__space_prototype.h          strange__space.h (generated)
  [defines types]                     [implements types]
        |                                    ^
        v                                    |
  parser + transformation  ──────────>  regenerate_space
        |                                    |
        └──── compiled against ──────────────┘
```

The parser and transformation are compiled against the current `strange__space.h`. When you regenerate, the meta program reads the prototype, parses it using the parser, transforms it using the transformation engine, and writes a new `strange__space.h`. The two-pass bootstrap then recompiles everything against the new output and regenerates a second time to verify stability.

## The Problem

When a new feature requires changes to both the prototype (adding a new field or type) and the parser or transformation (using that new field), you have a chicken-and-egg problem:

- The parser/transformation code references `abstraction.new_field()`, but that accessor doesn't exist yet in the current `strange__space.h`.
- The accessor won't exist until you regenerate from the updated prototype.
- But regeneration compiles the parser/transformation, which won't compile without the accessor.

## Techniques

### 1. Staggered Introduction

Break the change into phases that each leave the build in a compilable state.

**Phase 1: Add the data to the prototype and regenerate.**

Add the new member to the prototype (e.g., `bool hash {false};` in the `abstraction` struct). Update comparison operators if needed. Do not reference the new accessor in the parser or transformation yet. Regenerate -- this produces a new `strange__space.h` with the accessor available.

**Phase 2: Use the new accessor in the parser/transformation.**

Now that `abstraction.hash()` exists in the generated header, you can update `parse_abstraction_attribute()` to set it and `_hashes()` to read it. Regenerate again to verify.

**Phase 3: Add the attribute to prototype abstractions.**

Now that the parser understands `[[strange::hash]]`, add it to the prototype abstractions that need it. Regenerate.

### 2. Temporary Commenting

When a parser change must reference a new accessor that doesn't exist yet, you can temporarily comment out the line that uses it:

```cpp
// spc.includes().push_back(include);  // TODO: uncomment after regeneration
```

Regenerate to produce a `strange__space.h` with the new accessor, then uncomment and regenerate again.

### 3. Manual Generated File Fixup

When a change to the prototype produces generated code that doesn't compile against the old generated header (e.g., a parentheses mismatch in a comparison operator customisation), the bootstrap gets stuck: the checked-in `strange__space.h` has broken code, and regeneration compiles against it.

Fix the specific error directly in `strange__space.h`, then regenerate. The regeneration will overwrite your manual fix with correctly generated code.

Note that a failed regeneration may still have partially succeeded: pass 1 runs first and copies its output to source before pass 2 attempts to verify. If pass 1 produced correct output but pass 2 failed (e.g., due to a redefinition conflict), the source file already contains the pass 1 output. In this case, fix the remaining conflict (often in a non-generated file like `strange.h`) and regenerate again -- the bootstrap should now succeed on both passes.

### 4. Deferred Removal

When moving functionality from a manual location (e.g., `strange.h`) into the generated output:

1. Keep the old code in place while you regenerate with the new generation logic.
2. After regeneration succeeds and the generated header contains the new code, remove the old manual code.
3. Regenerate one final time to verify the two-pass bootstrap.

If the old and new code coexist temporarily, watch for redefinition errors on the second bootstrap pass (which compiles against the first pass output). If this happens, remove the old code and regenerate -- the first pass already produced the correct output.

## Ordering Summary

For a feature that adds a new prototype member, parser support, and transformation output:

| Step | Action | Regenerate? |
|------|--------|-------------|
| 1 | Add member to prototype, update comparison operators | Yes |
| 2 | Update parser to recognise the new attribute | No |
| 3 | Update transformation to emit new code | No |
| 4 | Add attribute to prototype abstractions | Yes |
| 5 | Remove old manual code (if migrating) | Yes |

Steps 2 and 3 can be done together without regenerating, as long as they don't reference accessors that require a prior regeneration (step 1 handles that).

## The Two-Pass Bootstrap

The `regenerate_space` CMake target runs:

1. **Pass 1**: Compile the meta generator against the checked-in `strange__space.h`, run it to produce a new header, copy to source.
2. **Pass 2**: Compile the meta generator against the pass 1 output, run it again, copy to source.
3. **Verify**: Compare hashes -- if pass 1 and pass 2 differ, the generated output is not stable and the build fails.

This means any change must produce stable output in two iterations. If pass 1 succeeds but pass 2 fails (e.g., due to redefinitions from deferred removal), the pass 1 output is already in source and you can fix the conflict and regenerate.

## Reducing Bootstrap Pain Through Auto-Generation

Hand-written `[[strange::customisation(...)]]` strings for comparison operators are a major source of bootstrap friction. Every new data member added to a prototype abstraction requires updating the `operator==` and `operator<` customisation strings -- adding fields to the `&&` chain and extending the nested parenthesised pattern for `operator<`. Getting the parenthesis count wrong in `operator<` (which requires exactly `2*(N-1) - 1` grouping parentheses for N fields) is an easy mistake that breaks the bootstrap and requires manual fixup of the generated file.

The `[[strange::equality]]`, `[[strange::comparison]]`, and `[[strange::hash]]` attributes eliminate this class of problem entirely. When the parser synthesizes comparison operators from data members, adding a new field to a prototype abstraction requires only adding the data member declaration -- the operators update themselves on the next regeneration. No customisation strings to maintain, no parentheses to count, and no bootstrap breakage from comparison operator typos.

Where possible, prefer auto-generation attributes over hand-written customisation strings for mechanical patterns that operate over all data members.
