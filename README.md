# strangen

A C++ metaprogramming framework for type erasure, reflection, code generation, and serialization.

> *"There is no exquisite beauty without some strangeness in the proportion."*
> -- Edgar Allan Poe

## Overview

Strangen generates type-erased C++ wrappers, reflection specializations, and closures from user-defined abstract type prototypes. You define types in a clean DSL-like prototype syntax, and strangen generates all the boilerplate for runtime polymorphism, reflection, serialization, copy-on-write semantics, weak references, and runtime type manufacturing.

The framework uses a multi-stage code generation pipeline: prototype definitions are parsed, transformed, and output as complete C++ headers ready for inclusion in application code. The library portion is header-only; only the `strangen` code generation tool itself needs to be compiled.

Strangen is self-hosting -- the core library's own 495KB `strange__space.h` header is generated by the same pipeline it provides to users.

## Features

- **Type Erasure** -- Runtime polymorphism via `_make<ConcreteType>()` factory pattern, without requiring inheritance in user types
- **Reflection** -- Runtime type names (`_name()`), categories (`_cat()`, `_cats()`), and type introspection via `strange::reflection<T>`
- **Code Generation** -- Two-stage meta-programming pipeline that transforms prototype definitions into full C++ headers
- **Prototype DSL** -- Define abstractions using C++ struct syntax with `[[strange::closure(...)]]` annotations
- **Closures** -- Generated callable wrappers (`std::function`) for abstraction methods
- **Serialization** -- JSON and binary serialization via the baggage system (powered by libdart)
- **Copy-on-Write** -- Shared pointer semantics with automatic clone-on-mutate via `_mutate()`
- **Weak References** -- `_weak()` / `_strong()` for breaking circular reference chains
- **Static/Dynamic Casting** -- `_static<T>()` and `_dynamic<T>()` for safe type conversions between abstractions
- **Multiple Inheritance** -- Diamond inheritance support with flattened category sets
- **Template Support** -- Full template parameters including variadics (`typename... Args`) with defaults
- **Manufacturing** -- Runtime type creation by registered name string via `_manufacture("type_name")`
- **Concurrency** -- Dataflow/reactive programming via stlab integration (graph and processor abstractions)

## Quick Start

### 1. Build the enstrange tool

```bash
cmake -B bake -DCMAKE_CXX_COMPILER=g++-12
cmake --build bake --target enstrange
```

### 2. Define a prototype

Create a prototype header that defines your abstractions. This file uses C++ struct syntax with special base classes and annotations:

```cpp
// my_space_prototype.h
#include "../../sauce/strange/meta/strange__space_prototype.h"
namespace demo
{
    struct food : strange::any
    {
        [[strange::closure("eat_closure_")]]
        auto eat(int xxx = 0) -> void;
    };

    struct fruit : food, strange::stuff
    {
        auto ripen() -> void;
        [[strange::closure("ripe_closure_")]]
        auto ripe() const -> bool;
        auto peel() -> void;
        auto peeled() const -> bool;
    };
}
```

Inherit from `strange::any` for basic type erasure, or from `strange::stuff` to add serialization support (`pack`/`unpack`).

### 3. Generate the space header

Run `enstrange` on the prototype to produce the generated header:

```bash
./bake/sauce/enstrange/enstrange my_space_prototype.h my_space.h
```

### 4. Build and run

Compile the output with `-I path/to/sauce/strange` to resolve the strange library includes. CMake handles this automatically for the demo:

```bash
cmake --build bake --target demo
./bake/demo
```

### 5. Use the generated types

Implement a concrete type and wrap it with the generated abstraction:

```cpp
#include "demo__space.h"

struct banana
{
    auto eat(int xxx) -> void { /* ... */ }
    auto ripen() -> void { _ripe = true; }
    auto ripe() const -> bool { return _ripe; }
    auto peel() -> void { _peeled = true; }
    auto peeled() const -> bool { return _peeled; }

    // Required for strange::stuff serialization support
    auto pack(strange::bag & dest) const -> void
    {
        dest.from_object();
        dest.insert_object("ripe", dest.make_bool(_ripe));
        dest.insert_object("peeled", dest.make_bool(_peeled));
    }

    auto unpack(strange::bag const & src) -> void
    {
        src.get_object("ripe").as_bool(_ripe);
        src.get_object("peeled").as_bool(_peeled);
    }

private:
    bool _ripe = false;
    bool _peeled = false;
};

// Register the type name for reflection and manufacturing
namespace strange
{
    template<>
    struct reflection<banana>
    {
        static inline auto name() -> std::string { return "banana"; }
    };
}

int main()
{
    // Create a type-erased fruit from a concrete banana
    auto fruit_1 = demo::fruit::_make<banana>();

    // Call operations through the type-erased interface
    fruit_1.ripen();
    fruit_1.ripe();

    // Generate and use closures
    auto closure = fruit_1.ripe_closure_();
    closure(); // calls ripe() through the closure

    // Serialize to JSON
    auto bag = strange::baggage::_make();
    bag.from_any(fruit_1._static<strange::any>());
    std::cout << bag.to_json();

    // Manufacture a new instance by registered type name
    auto fruit_2 = demo::fruit::_manufacture("demo::fruit_<banana, true>");
}
```

## Code Generation Pipeline

```
Prototype (.h)              enstrange tool              Generated Space Header (.h)
[DSL definitions]  ------>  [parse + transform]  ------>  [type erasure wrappers,
                                                           reflection, closures]
```

### Stage 1: Prototype File

Define abstractions using C++ struct syntax. The prototype must include `strange__space_prototype.h` and place abstractions in a namespace:

- Inherit from `strange::any` for basic type erasure
- Inherit from `strange::stuff` to add `pack`/`unpack` serialization support
- Use `[[strange::closure("name")]]` to generate callable wrappers
- Use `[[strange::implementation("name")]]` to generate implementation structs with data storage
- Use `[[strange::hash]]`, `[[strange::equality]]`, `[[strange::comparison]]` to auto-generate boilerplate
- Template abstractions support defaults and variadics
- Multiple inheritance is supported (e.g., `struct bunch_of_fruit : food, bunch<fruit>, strange::stuff`)

### Stage 2: Run enstrange

```bash
./bake/sauce/enstrange/enstrange my_space_prototype.h my_space.h
```

The `enstrange` tool reads the prototype, parses it, applies transformations, and writes the complete type erasure implementation to the output file.

### Stage 3: Use Generated Code

Include the generated space header in your application code (after `#include "strange.h"`) and compile with the required dependency include paths.

## Prototype DSL Reference

### Basic Abstraction

```cpp
struct widget : strange::any
{
    auto display() const -> void;
    auto inc() -> void;
};
```

### Serializable Abstraction

Inheriting from `strange::stuff` adds `pack` and `unpack` operations:

```cpp
struct fruit : food, strange::stuff
{
    auto ripen() -> void;
};
```

### Abstraction Attributes

Abstraction attributes appear as `[[strange::...]]` before the struct name. Multiple attributes can be stacked.

#### `thing("name")`

Sets the default concrete type for `_make()`. Used when the implementation type is external (not generated):

```cpp
struct [[strange::thing("std::vector<T>")]]
vector : strange::stuff
{
    auto push_back(T const & item) -> void;
};
```

Callers can then write `auto v = my_namespace::vector<int>::_make();` without specifying the concrete type -- it defaults to `std::vector<T>`.

#### `implementation("name")`

Generates an implementation struct with automatic data member storage, inline accessors, and `pack`/`unpack` serialization for `stuff` derivatives. If `thing` is not specified, it is auto-set to the same value:

```cpp
struct [[strange::implementation("my_namespace::impl::widget")]]
[[strange::hash]]
[[strange::equality]]
[[strange::comparison]]
widget : strange::stuff
{
    std::string name {};
    int64_t value {};
};
```

This generates a `my_namespace::impl::widget` struct with `name_` and `value_` member variables, const/mutable accessors, and `pack`/`unpack` methods. The generated struct becomes the default `_Thing` for `_make()`.

#### `hash`

Generates a `std::hash` specialization for the abstraction, hashing all data members. Vector members use `strange::hash_range`; scalars use `strange::hash_init`/`strange::hash_combine` (defined in `strange__hash.h`). Requires `implementation`.

#### `equality`

Generates `operator==` (checks all data members with `&&`) and `operator!=` (delegates to `==`). Requires `implementation`.

#### `comparison`

Generates `operator<` (lexicographic ordering over all data members), `operator<=` (delegates to `<` and `==`), `operator>` (delegates to `<=`), and `operator>=` (delegates to `<`). Requires `implementation`.

### Operation Attributes

Operation attributes appear as `[[strange::...]]` before an operation declaration.

#### `closure("name")`

Generates a companion method that returns a `std::function` wrapping the operation:

```cpp
struct food : strange::any
{
    [[strange::closure("eat_closure_")]]
    auto eat(int xxx = 0) -> void;
};
```

This generates an `eat_closure_()` method returning `std::function<void(int)>`.

#### `customisation("code")`

Replaces the default forwarding body in the inner (instance) class. Normally, the instance class forwards calls to the concrete object via `_thing.method(args)`. A customisation replaces this with custom code:

```cpp
[[strange::customisation("return _thing == other.template _static<widget_<_Thing, _Copy> const>()._thing()")]]
auto operator==(widget const & other) const -> bool;
```

This is used when the default forwarding is insufficient -- for example, comparing the underlying objects of two type-erased wrappers requires extracting `_thing` from both sides.

#### `modification("code")`

Replaces the forwarding code in the type-erased wrapper class. Normally, the wrapper calls `_mutate()` (for non-const operations) then forwards via `dynamic_pointer_cast<_derived>(shared)->method(args)`. A modification replaces this entire sequence:

```cpp
[[strange::modification("auto const index = pos - cbegin();"
"    strange::_common::_mutate();"
"    pos = cbegin() + index;"
"    return std::dynamic_pointer_cast<typename vector<T>::_derived>"
"(strange::_common::_shared)->insert(pos, value);")]]
auto insert(random_access_const_iterator<T> pos, T const & value) -> random_access_iterator<T>;
```

This is used when `_mutate()` invalidates state (like iterator positions) that must be saved and restored around the clone.

#### `result("type")`

Overrides the return type parsed from the operation signature. The special values `"*this"` and `"*that"` generate return-self semantics:

```cpp
[[strange::result("*this")]]
auto operator++() -> forward_const_iterator &;

[[strange::result("*that")]]
auto operator++(int i) -> forward_const_iterator;
```

`"*this"` returns a reference to self after the operation (prefix increment). `"*that"` returns a copy of self taken before the operation (postfix increment).

### Inline Operation Implementations

Operations can include an inline implementation between the return type and the terminating semicolon. This provides the method body for the generated implementation struct:

```cpp
struct [[strange::implementation("my_namespace::impl::widget")]]
widget : strange::any
{
    std::string name {};
    auto compute() const -> int { return name().length(); };
};
```

The implementation struct will contain `inline auto compute() const -> int;` declared in the struct, with the definition `{ return name().length(); }` emitted at the end of the generated output. Data members automatically get inline accessors (`{ return name_; }`) without needing explicit implementations.

### Template Abstractions

```cpp
template<typename Item = int, typename... Args>
struct bunch : strange::any
{
    auto push_back(Item const & item) -> void;
    auto size() const -> std::size_t;
    auto operator[](std::size_t pos) const -> Item const &;
    auto operator[](std::size_t pos) -> Item &;
};
```

### Multiple Inheritance

```cpp
struct bunch_of_fruit : food, bunch<fruit>, strange::stuff
{
    using Item = fruit;
    auto pick() -> void;
    auto picked() const -> bool;
};
```

## Using Generated Types

### Construction

```cpp
// Create a type-erased wrapper around a concrete implementation
auto fruit = demo::fruit::_make<banana>();

// Create with constructor arguments
auto nb = demo::fruit::_make<no_banana>(std::string{});

// Create by registered type name at runtime
auto fruit = demo::fruit::_manufacture("demo::fruit_<banana, true>");

// Default construction creates an empty (invalid) instance
demo::fruit empty;
```

### Introspection

```cpp
fruit._something();  // true if the instance holds a value
fruit._name();       // runtime type name (e.g., "demo::fruit_<banana, true>")
fruit._cat();        // category name (e.g., "demo::fruit")
fruit._cats();       // set of all categories (all implemented abstractions)
fruit._copy();       // whether the underlying object is copyable
fruit._address();    // address of the underlying concrete object
fruit._sizeof();     // size of the underlying concrete object
fruit._error();      // get error message (empty if no error)
fruit._error("msg"); // set error state with message
```

### Casting

```cpp
// Unchecked cast to another abstraction (like static_cast)
auto any = fruit._static<strange::any>();

// Checked cast, returns empty if the types don't match (like dynamic_cast)
auto food = fruit._dynamic<demo::food>();
if (food._something())
{
    food.eat();
}
```

### Copy-on-Write

```cpp
auto fruit_1 = demo::fruit::_make<banana>();
auto fruit_2 = fruit_1; // shared pointer, same underlying object

fruit_1._mutate(); // clones the underlying object if shared
// fruit_1 and fruit_2 now point to different objects
```

### Weak References

```cpp
auto weak = fruit._weak();   // create a weak reference
auto strong = weak._strong(); // resolve back to a strong reference (may be empty)
fruit._is_weak();             // check if this is a weak reference
```

### Closures

For operations annotated with `[[strange::closure("name")]]`:

```cpp
auto fruit = demo::fruit::_make<banana>();
auto closure = fruit.eat_closure_();  // returns std::function<void(int)>
closure(0); // calls eat(0) on the fruit

auto ripe = fruit.ripe_closure_();    // returns std::function<bool()>
bool is_ripe = ripe();
```

### Implementing a Concrete Type

A concrete type is a plain struct that implements the same method signatures as the abstraction:

```cpp
struct banana
{
    // Match the abstraction's operation signatures
    auto eat(int xxx) -> void { /* ... */ }
    auto ripen() -> void { _ripe = true; }
    auto ripe() const -> bool { return _ripe; }
    auto peel() -> void { _peeled = true; }
    auto peeled() const -> bool { return _peeled; }

    // For strange::stuff support, implement pack/unpack
    auto pack(strange::bag & dest) const -> void
    {
        dest.from_object();
        dest.insert_object("ripe", dest.make_bool(_ripe));
    }

    auto unpack(strange::bag const & src) -> void
    {
        src.get_object("ripe").as_bool(_ripe);
    }

private:
    bool _ripe = false;
    bool _peeled = false;
};

// Register for reflection and manufacturing
namespace strange
{
    template<>
    struct reflection<banana>
    {
        static inline auto name() -> std::string { return "banana"; }
    };
}
```

Non-copyable types are supported -- delete the copy constructor and the framework handles it:

```cpp
struct no_banana
{
    no_banana(std::string x) {}
    no_banana(no_banana const &) = delete;
    // ... operations ...
};
```

## Serialization (Baggage)

The baggage system provides JSON and binary serialization for type-erased objects. It is powered by the libdart library.

### API

```cpp
auto bag = strange::baggage::_make();

// Serialize from a type-erased object
bag.from_any(fruit._static<strange::any>());

// Deserialize back
auto restored = bag.to_any()._static<demo::fruit>();

// JSON conversion
std::string json = bag.to_json();
bag.from_json(json);

// Binary conversion
auto binary = bag.to_binary();
bag.from_binary(binary);
```

### Primitive Values

```cpp
auto bag = strange::baggage::_make();
bag.from_bool(true);
bag.from_int64(-123);
bag.from_uint64(456);
bag.from_double(789.1);
bag.from_string("hello");
bag.from_array();
bag.from_object();
bag.from_null();
```

### Sealing

Bags can be sealed (finalized) for immutable binary representation:

```cpp
bag.seal();     // finalize
bag.sealed();   // check if sealed
bag.unseal();   // make mutable again
```

### Concrete Type Requirements

To support serialization, concrete types must implement `pack` and `unpack`, and inherit from `strange::stuff` in the prototype:

```cpp
auto pack(strange::bag & dest) const -> void
{
    dest.from_object();
    dest.insert_object("key", dest.make_bool(value));
    // For arrays of type-erased objects:
    auto arr = dest.make_array();
    for (auto const & item : items)
    {
        arr.push_back_array(dest.make_any(item._static<strange::any>()));
    }
    dest.insert_object("items", arr);
}

auto unpack(strange::bag const & src) -> void
{
    src.get_object("key").as_bool(value);
    auto arr = src.get_object("items").to_array();
    // ... restore items ...
}
```

## Concurrency (Graph/Processor)

The concurrency module provides dataflow programming via stlab channel and processor abstractions. Requires the stlab library.

### Channels

```cpp
#include "../../sauce/strange/implementation/graph/strange__implementation__graph.h"

stlab::sender<int> send;
stlab::receiver<int> receive;
std::tie(send, receive) = stlab::channel<int>(stlab::high_executor);

auto discard = receive | [](int x) {
    std::cout << "received: " << x << "\n";
};
receive.set_ready();
send(42);
```

### Processors and Graphs

Processors and graphs are parameterised by `Config` (configuration type passed to `closure()`) and `Signal` (the data type flowing through the graph). Port counts and types are set at construction time via type vectors -- processors and graphs are immutable after construction.

```cpp
// Create a graph with 3 input ports and 3 output ports (all type 0)
auto graph = strange::graph<std::string, std::string>::_make(
    std::vector<uint64_t>{0, 0, 0},   // input types
    std::vector<uint64_t>{0, 0, 0});  // output types

// Create a processor with 1 input and 1 output (type 0)
auto proc = strange::processor<std::string, std::string>::_make<
    strange::implementation::thru_processor<std::string, std::string>>(
    std::vector<uint64_t>{0});
auto proc_id = graph.add_processor(proc);

// Connect ports (from_id, from_out, to_id, to_in)
// id 1 = input node, id 0 = output node
graph.add_connection(make_connection({
    .from_id_ = 1, .from_out_ = 0,
    .to_id_ = proc_id, .to_in_ = 0}));
graph.add_connection(make_connection({
    .from_id_ = proc_id, .from_out_ = 0,
    .to_id_ = 0, .to_in_ = 0}));

// Execute the graph
auto graph_closure = graph.closure();
std::vector<std::string> inputs {"hello", "world", "!"};
auto outputs = graph_closure(inputs);
```

Type vectors specify the type identifier for each port. `add_connection` validates that connected ports have matching types. `thru_processor` uses a single type vector (input and output ports share the same types). Subgraphs can be nested inside larger graphs, and graphs support split, join, and pass-through connection patterns.

## Architecture

This section covers the internals for contributors who want to understand or modify the framework.

### Naming Conventions

- **Files** use double-underscore separators: `strange__common.h`, `strange__space.h`
- **Generated types** produce a companion template with a trailing underscore: `food` generates `food_<_Thing, _Copy>`
- **Internal methods** are prefixed with an underscore: `_make`, `_valid`, `_mutate`, `_static`, `_dynamic`

### The `_common` / `_base` Pattern

All strange abstractions share a common architecture:

- **`_common`** is the non-virtual base class for all abstractions. It holds a single `std::shared_ptr<_base>` and provides boilerplate for casting, copy-on-write, reflection, and error handling. `_common` itself has no virtual functions or v-table.

- **`_base`** is the virtual interface with pure virtual methods for cloning (`_clone`), reproduction (`_reproduce`), type info (`_cat`, `_cats`, `_name`, `_copy`), and memory inspection (`_address`, `_sizeof`).

- **`_derived`** is generated per-abstraction, extending `_base` with pure virtual methods for that abstraction's operations (e.g., `virtual auto eat(int) -> void = 0`).

- **`_instance`** is the innermost concrete class inside the generated `food_<_Thing, _Copy>` template. It holds a `_Thing` instance and forwards all virtual calls to it.

```
_common (non-virtual, holds shared_ptr<_base>)
  └── _base (virtual: _clone, _name, _cat, ...)
        └── _derived (virtual: eat, ripen, ripe, ...)  [generated per-abstraction]
              └── _instance (final: wraps _Thing)       [generated per-abstraction]
```

### The strangen Preprocessor

The `strange__generation.cpp` tool is a character-level state machine that processes two kinds of delimited blocks:

- **`/*~ ... */`** -- block meta code (content between `/*~` and `*/` becomes executable C++)
- **`//~ ...`** -- line meta code (content after `//~` until end of line becomes executable C++)

Everything outside these delimiters becomes raw string literals (`R"~(...)~"`) written to stdout via `std::cout`. The output is a complete `main()` function that, when compiled and run, produces the final header file.

### Parser and Transformation Engine

- **Tokenizer** (`strange__comprehension__toker.h`) -- Reads characters from a prototype file and produces tokens classified as: name, number, character, string, comment, punctuation, whitespace, or mistake.

- **Parser** (`strange__comprehension__parser.h`) -- Parses tokens into structured `strange::space`, `strange::abstraction`, `strange::operation`, and `strange::parameter` objects. Handles `#include` directives for recursive composition of spaces.

- **Transformation** (`strange__transformation.h`) -- Takes a parsed `strange::space` and outputs complete C++ code including: forward declarations, `strange::reflection<>` specializations, abstraction structs with `_derived` inner classes, implementation templates (`_<_Thing, _Copy>`), and factory registration for `_manufacture`.

### Self-Hosting

The core library's own `strange__space.h` (~495KB) is generated by the same pipeline:

1. `sauce/strange/meta/strange__space_prototype.h` defines the core abstractions (`any`, `stuff`, `bag`, `space`, `abstraction`, `operation`, `parameter`, `token`, etc.)
2. The CMake `regenerate_space` target compiles `enstrange` from source and runs a two-pass bootstrap: hashes the checked-in `strange__space.h`, regenerates it twice (pass 2 recompiles `enstrange` against pass 1 output), copies results to source, and fails the build if the hash changed

## Examples

### Baggage (`snacks/baggage/`)

Demonstrates JSON and binary serialization using the libdart library and the strange baggage system, including round-trip serialization of parsed space definitions.

```bash
cmake --build bake --target baggage && ./bake/snacks/baggage
```

### Currants (`snacks/currants/`)

Demonstrates dataflow programming with stlab channels (split, join, zip), custom process functors, processor graphs, and the strange graph abstraction. Requires stlab (built automatically by CMake if available).

```bash
cmake --build bake --target currants && ./bake/snacks/currants
```

### Demo (`snacks/demo/`)

Full-featured demonstration including type erasure, closures, templates, multiple inheritance, serialization, and runtime manufacturing with a fruit/banana type hierarchy.

```bash
cmake --build bake --target demo && ./bake/snacks/demo
```

### Example (`snacks/example/`)

**Note:** This example is outdated and may be misleading. It uses the older `strangen` preprocessor pipeline and programmatic space construction rather than the current `enstrange`-based prototype approach. Refer to the demo for up-to-date usage patterns.

### Running Examples

After building, use `snacks.sh` to run examples from both compiler builds:

```bash
bash snacks.sh            # run all examples quietly (pass/fail)
bash snacks.sh purge      # run all examples with full output
bash snacks.sh baggage    # run a single example with output
```

Available snack names: `baggage`, `currants`, `demo`, `example`. Currants is skipped for clang builds (stlab dependency).
